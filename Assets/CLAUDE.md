## Project overview

A Unity 6 URP **2D liquid simulation** system. The shippable product is the UPM package
`com.blurfeng.liquid-2d-simulation` under `Assets/Plugins/Liquid2DSimulation/` (namespace root
`Fs.Liquid2D`, asmdef `Liquid2DSimulation`). The surrounding Unity project at the repo root is a
**development/demo harness** for that package — its scenes, `Assets/Source/`, and settings exist to
exercise and demo the plugin, not to ship.

- Engine: Unity `6000.3` (see `ProjectSettings/ProjectVersion.txt`); package min is `6000.2`.
- Branches: `main` targets Unity 6 (Render Graph); a `2022.3` branch supports older Unity and lags behind.
- Package version lives in `Assets/Plugins/Liquid2DSimulation/package.json` (depends on URP `17.2.0`).

## Build / run / test

- **No CLI build.** Open the project in the Unity Editor (`6000.3`). Compilation = the editor
  compiling on focus; "build verification" means opening in Unity and checking the Console.
- **Run/iterate**: enter Play mode on a dev scene in `Assets/Scenes/` (e.g. `DevScene/DevScene.unity`,
  `2 WaterMix`, `3 Milk`, `4 Minisize`). The package `Samples~/` scenes mirror these.
- **`.sln` / `*.csproj` are Unity-generated** and regenerated by the editor — never hand-edit them.
- **Tests**: there is no project test suite or Tests asmdef. If adding tests, create a new Tests asmdef.
- **Releasing the package** = the `Assets/Plugins/Liquid2DSimulation` subtree only. The UPM git URL
  pins it via `?path=Assets/Plugins/Liquid2DSimulation`. Bump the version in `package.json`.

## Architecture

There are **two decoupled halves**: a **physics simulation** that moves particles, and a **URP Render
Graph pipeline** that draws the fluid look. They meet only at a data hand-off — the renderer reads particle
state out of the simulation, never off Transforms.

### Physics — pure-data SPH (`Runtime/Source/Systems/Physics/`)

There are **no per-particle GameObjects**. Particles are rows in Structure-of-Arrays buffers, advanced by an SPH **dual-density** solver.

- **`Liquid2DSimulation.cs`** — runtime singleton hub (`[DefaultExecutionOrder(-100)]`, lazily creates a
  hidden GameObject; **play-mode only**). Owns the `Liquid2DParticleStore`, the solver, the descriptor /
  material tables, and per-`nameTag` groups. Drives solving in **`FixedUpdate`**. Public API: `Spawn` /
  `Despawn` / `GetPosition` / `GetVelocity` / `IsAlive` / `SetColor`. Static config knobs: `Params`,
  `Mode`, `MaxParticlesPerTag`, `GpuReadbackToStore`.
- **`Liquid2DParticleStore.cs`** — SoA store (`positions`, `velocities`, `colors`, `radii`, `typeId`,
  `groupId`, `lifetimeEnd`, `alive`, …) as parallel `NativeArray`s for Burst. **Slots are stable**:
  a handle's `Index` == its slot; freeing sets `alive=0` and bumps a `version` for staleness detection.
  Each step `Liquid2DSimulation` rebuilds a compact active-index list skipping holes.
- **`ILiquid2DSolver` + `SphCpuSolver` / `SphGpuSolver`** (`Solvers/`) — the **CPU/GPU seam**. CPU =
  Job System + Burst (`SphJobs.cs`, `Liquid2DHashGrid.cs`); GPU = compute shader `Resources/Liquid2DSph`
  with **GPU-resident** buffers (data stays on the GPU; spawns upload incrementally via `gpuPendingSpawns`).
  Mode is chosen by `Liquid2DSimulation.Mode`; GPU falls back to CPU if compute shaders are unsupported.
- **`Liquid2DParticleDescriptor.cs`** — `ScriptableObject` that **replaces the particle prefab** in the
  pure-data model: defines a particle *type*'s `radius`, `renderScale`, lifetime, `renderSettings`
  (sprite/material/color/`nameTag`), `mixSettings`, and physics `material`. Registered into the sim to get a
  `typeId`; rendering batches by descriptor.
- **Colliders** (`Physics/Colliders/`) — these *stay* MonoBehaviours (few, scene-authored):
  `Liquid2DCollider` subclasses (Box/Circle/Capsule/Polygon) register into `Liquid2DColliderRegistry`,
  flattened to a Burst buffer each frame. `isDynamic` colliders receive fluid reaction impulse (two-way
  coupling) via `ILiquid2DForceReceiver`.
- **`Liquid2DPhysicsConfig.cs`** (in `Systems/`) — optional scene component that pushes solver params,
  compute mode, cap, and fixed timestep into the static `Liquid2DSimulation` fields on `Awake`.

### Rendering — URP Render Graph (`Runtime/Source/Systems/`)

- **`Liquid2DFeature.cs`** — `ScriptableRendererFeature` added to the URP 2D Renderer Data; creates the pass.
- **`Liquid2DPass.cs`** — `ScriptableRenderPass` at `RenderPassEvent.AfterRenderingTransparents`. Per frame:
  grab camera color as background (`GrabAsBg`) → draw particles (reads sim data, not Transforms) → iterative
  blur with a "core keep" overlay (approximate SDF fusion) → render obstructor/occluder Rendering-Layer
  textures → composite via the `Liquid2DEffect` material (cutoff, opacity, cover color, edge, pixel, distort
  toggled through **shader keywords**).
- **`Liquid2DVolume.cs`** (in `Volumes/`) — URP Volume overriding feature settings at runtime;
  `Liquid2DPass.UpdateSettings()` merges Volume overrides over `Liquid2DRenderFeatureSettings` each frame.
  Matched to a feature by **`nameTag`**.
- **Render data hand-off**: `Liquid2DSimulation.TryGetRenderData(...)` (CPU store + active indices) and
  `TryGetRenderBuffers(...)` (resident GPU `ComputeBuffer`s for `DrawProcedural`). Safe to read at render
  time because solve jobs are completed inside `FixedUpdate`.
- **`Liquid2DParticleDisplay.cs`** (in `Systems/`) — **test-only** standalone visualizer. Draws active
  particles with `Graphics.DrawProcedural` (quad generated in-shader) straight from the sim, bypassing the
  Feature; shader `Custom/URP/2D/Liquid2DParticleDisplay`. Use it to debug the sim independent of the full
  effect pipeline; the *shipping* path is the Feature.

Supporting pieces:

- **Spawning**: `Systems/Gameplay/LiquidSpawner.cs` picks a weighted-random `Liquid2DParticleConfig` (which
  now wraps a `Liquid2DParticleDescriptor`, not a prefab) and calls `Liquid2DSimulation.Instance.Spawn`,
  applying flow rate, eject force, swing, and lifetime.
- **`Loader.cs`** — static façade over instantiate/destroy with injectable delegates (`InitLoaderWith*`,
  `InitDestroy`); default = `Resources.Load` + `Object.Instantiate`/`Destroy`. The **pooling seam**:
  `Assets/Source/GameInstance.cs` routes it through an object `Pool`. (Now used for non-particle objects;
  particles themselves are pure-data and never instantiated.)
- **Shaders**: `Runtime/Source/Shaders/` — `Liquid2DParticle`(`Gpu`), `Liquid2DParticleDisplay`,
  `Liquid2DBlur`, `Liquid2DEffect`, plus `CombineTwo`/`GrabAsBg`/`Clone` blit helpers; resolved by
  `Shader.Find("Custom/URP/2D/...")`.

Scene wiring: add `Liquid2DFeature` to the URP 2D Renderer Data, and configure **Rendering Layer Masks**
(not sorting layers) for obstructors (block flow) and occluders (cover but don't block).

## Conventions / gotchas

- **Trilingual docs**: public API XML comments are written in Chinese / English / Japanese, and inspector
  tooltips use the custom `LocalizationTooltipAttribute(zh, en, ja)` (picks language by system locale).
  Match this style when editing existing code.
- **`nameTag` links particles ↔ feature ↔ volume** (and also groups particles in the sim store). An empty
  `nameTag` is rendered by every feature; a set tag restricts it to the matching feature/volume.
- **Physics is play-mode only.** `Liquid2DSimulation.Instance` returns `null` in edit mode / on quit, and
  the sim runs on `FixedUpdate`. Particles no longer appear in the Scene view via `[ExecuteAlways]`.
- **GPU mode keeps particle data on the GPU.** Under `Mode == Gpu` the CPU `store` is **stale** except on
  buffer-grow frames, so `GetPosition`/`GetVelocity`/`Liquid2DDebugGizmos` are unreliable. Set
  `gpuReadbackToStore` (on `Liquid2DPhysicsConfig` / `Liquid2DSimulation.GpuReadbackToStore`) to force a
  per-frame readback — but it's a **synchronous GPU→CPU stall**; keep it off for production. The Feature and
  `Liquid2DParticleDisplay` read GPU buffers directly and are unaffected.
- **Burst/native cleanup**: `NativeArray`/`ComputeBuffer` allocations must be disposed (see the `try/finally`
  around `_solver.Step` and `OnDestroy` in `Liquid2DSimulation.cs`). TempJob buffers are freed each frame.
- **Rendering Layer Mask caveat**: layer-mask config doesn't survive importing the plugin into a fresh
  project (engine caching makes the demo look fine while the layers don't actually exist). See README.
- Make user-facing changes inside `Assets/Plugins/Liquid2DSimulation/`, and keep the package `Samples~/`
  scenes in sync with the dev `Assets/Scenes/`.
- Editor-only code is guarded by `#if UNITY_EDITOR`.
- Full usage/configuration docs: `README.md` (中文), `README_EN.md`, `README_JA.md`.

## Unity coding conventions

- **Inspector fields stay private.** Any field that is serialized and exposed for editing in the Inspector
  is always declared `private` with `[SerializeField]` — never `public`. This applies to every serializable
  context: `MonoBehaviour`, `ScriptableObject`, and plain `[Serializable]` data classes/structs nested in
  them. When such a field must be read or written from outside the class, expose a dedicated accessor
  property instead of widening the field's visibility.
